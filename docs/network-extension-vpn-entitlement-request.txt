Network Extension / VPN Entitlement Request - Nima

Plain English summary:
Nima is an iOS app that helps users block distracting short-form feeds, such as Instagram Reels and TikTok video feeds, while preserving useful app functions like messaging where possible. Nima uses an on-device Network Extension packet tunnel so the user can enable local blocking rules on their own device. The VPN is not used for advertising, tracking, resale of network data, or third-party traffic analytics.

What the VPN does:
Nima creates a local VPN configuration controlled by the user. The host app uses NETunnelProviderManager and NETunnelProviderProtocol to create, save, load, start, and stop the VPN profile. The tunnel provider is implemented with NEPacketTunnelProvider.

When the tunnel starts, the extension:
- Creates packet tunnel network settings.
- Sets an IPv4 tunnel route.
- Applies DNS settings.
- Starts a local SOCKS5 proxy inside the extension.
- Uses tun2socks to route packet tunnel traffic through the local proxy path.
- Applies local blocking rules selected by the user.

Why Network Extension is required:
Nima needs packet-level access through Apple's Network Extension framework to enforce blocking rules across selected app traffic on iOS. Standard in-app networking APIs cannot block traffic generated by other apps, and Screen Time-style app blocking is too broad for Nima's purpose because Nima is designed to block distracting feed/video surfaces while preserving useful app functions where technically possible.

Primary user benefit:
The user can reduce compulsive scrolling without fully deleting or blocking entire social apps. For example, a user may choose to block short-form video/feed behavior while still allowing useful messaging or account functions when possible.

Use case category:
- Content blocking
- Digital wellbeing / distraction reduction
- User-controlled local filtering

What traffic is processed:
The Network Extension processes only the network signals needed to apply the user's blocking rules and maintain the tunnel. These may include:
- Domain names and hostnames
- SNI values when visible
- IP addresses and ports
- Connection metadata
- Byte counts used for local classification and blocking thresholds
- Local block/allow decisions
- VPN lifecycle diagnostics such as start, stop, reconnect, and error state

What traffic is not processed:
- Nima does not read message content.
- Nima does not read passwords.
- Nima does not read photos.
- Nima does not inspect private app UI content.
- Nima does not decrypt HTTPS traffic in the App Store iOS app.
- Nima does not install a root CA certificate for traffic interception in the App Store iOS app.
- Nima does not use the VPN to build advertising profiles.
- Nima does not use the VPN to track users across apps or websites.

Blocking behavior:
Nima classifies traffic locally using domains, SNI where available, ports, connection metadata, byte counts, and app-specific heuristics. The user controls which blocking policies are active. The tunnel may block or terminate traffic that matches enabled feed/video blocking rules, while allowing traffic that appears to be control or messaging traffic where technically possible.

Examples of supported blocking goals:
- Block Instagram Reels or short-form media surfaces.
- Block TikTok video feed traffic.
- Block X/Twitter feed media or feed API traffic depending on user-selected settings.
- Preserve useful non-feed behavior where the local classifier can distinguish it.

User consent and control:
- Nima presents an in-app explanation before requesting VPN permission.
- Nima explains that it uses a local VPN configuration to block selected feeds.
- Nima explains that it processes network signals such as domains, connection metadata, and block decisions to apply the user's rules.
- The user must approve the iOS VPN configuration prompt.
- The user can turn blocking/VPN behavior on or off from inside the app.
- The user can remove or disable the VPN configuration in iOS Settings.

Data collection and privacy:
Nima's VPN-derived network signals are processed locally for blocking. Ordinary browsing metadata is not transmitted to Supabase, OpenAI, analytics providers, advertising providers, or other third parties.

Local diagnostics:
Some traffic details may appear in local diagnostic files on the user's device, such as tunnel logs, connection errors, block decisions, VPN lifecycle state, and performance counters. These diagnostics are used for troubleshooting and are stored locally unless the user chooses to share them with support.

Support diagnostics:
If a user contacts support and chooses to share a diagnostic report, Nima may receive the diagnostic information the user provides. This is used only for support, reliability troubleshooting, and app functionality.

Third-party processors:
- Supabase: authentication, sessions, profile/blocker state, waitlist, account deletion, and cancellation feedback.
- RevenueCat: subscription status, purchase history, entitlements, offerings, purchases, and restores.
- Google Sign-In: Google authentication data.
- Apple: Sign in with Apple and App Store purchase processing.
- Hosting/infrastructure providers: limited app, website, database, and service operations.

VPN data sharing commitment:
Nima does not sell, use, or disclose data derived from VPN use to third parties for any purpose. Nima does not use VPN-derived data for third-party advertising, cross-app tracking, profiling, or data resale.

Tracking and advertising:
- Nima does not use IDFA.
- Nima does not use App Tracking Transparency because it does not track users across apps or websites owned by other companies.
- Nima does not include third-party advertising SDKs for behavioral advertising.
- Nima does not sell personal data.
